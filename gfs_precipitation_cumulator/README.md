## Overview

The `gfs_precipitation_cumulator` processor calculates accumulated precipitation over specified time periods using Global Forecast System (GFS) hourly data. It acts as an orchestrator, automatically determining the required data, triggering the `gfs_fetcher` app to retrieve it, and then securely summing the hourly files into distinct cumulation maps (e.g., 1hr, 3hr, 6hr, 12hr, 24hr) ending at a specific target time. 

### Key Features

  * **Dynamic Orchestration**: Automatically calculates the exact initialization date and fetch duration required to reach your target time (seamlessly scaling up to 120 hours / 5 days), safely clamping future dates, and triggers the `gfs_fetcher` app seamlessly in the background.
  * **Historical Hindcasting**: Features a dedicated `HINDCASTING_MODE` that overrides live operational constraints, allowing you to explicitly target past model runs and specific historical dates to reconstruct past precipitation events. *(Note: Currently, only historical GFS files from **2021-01-01 onwards** are available in the archive for hindcasting).*
  * **Unit Conversion**: Automatically multiplies the raw GFS Precipitation Rate (PRATE) data by 3600, converting it from mm/s to total accumulated millimeters (mm).
  * **Grid Alignment**: Automatically snaps the user-defined Bounding Box to match the native GFS 0.25° grid, ensuring accurate spatial processing.

### Output Maps

The processor generates a series of accumulated GeoTIFF files, one for each requested cumulation period.

  * Filename Convention: Output files are named using the base name, the cumulation period, and the start and end valid times of the summation.
  * The format is: `{BaseName}_Cumulative_{Period}_GFS_{StartTime}-{EndTime}.tif`
  * Example: `MyArea_Cumulative_24hr_GFS_20260318_1600-20260319_1600.tif` (This represents a 24-hour accumulation ending on March 19, 2026, at 16:00).
  * Legend: Pixel values represent the total accumulated precipitation in millimeters (mm).
  * Data Type: Float32.
  * Payload: The final job payload includes the original input parameters, the list of intermediate GFS files used (`GFS_FILES`), the final processed output GeoTIFF file names (`OUTPUTS`), and the total count of generated images.

### Parameters

#### Basic
  * `BASE_NAME`: A base name or prefix for all output files (it is recommended to use "-" instead of "\_" or spaces).
  * `BBOX`: The Bounding Box that defines your Area of Interest. The app will automatically align this to the nearest 0.25° GFS grid coordinates.
  * `TARGET_DATETIME` (optional): The precise end time for your cumulations, in "YYYY-MM-DD HH:MM" format. If left blank, it defaults to the current UTC hour.

#### Advanced
  * `CUMULATION_PERIODS` (default is "1hr,3hr,6hr,12hr,24hr"): A comma-separated list of the hourly accumulation windows you want to generate.
  * `RUN_TYPE` (default is LAST): The GFS model run to pass down to the fetcher. `LAST` will automatically find the most recent available run. Specific runs can be requested with "00", "06", "12", or "18".
  * `HINDCASTING_MODE` (default is false): Set to `true` to disable dynamic date clamping. This allows the processor to fetch and cumulate data from past historical model runs.
  * `SEARCH_DATE` (optional): Required if `HINDCASTING_MODE` is `true`. Specifies the exact historical initialization date to fetch (in "YYYY-MM-DD" format). **Important: Must be 2021-01-01 or later.**
  * `DELETE` (default is true): Set to `false` to keep the intermediate hourly files generated by the fetcher in your active workspace after the cumulation is complete.

### JSON Sample

```json
{
    "BASE_NAME": "MyArea",
    "BBOX": {
      "northEast": {
        "lat": 15.3,
        "lng": -16.6
      },
      "southWest": {
        "lat": 14.3,
        "lng": -17.6
      }
    },
    "TARGET_DATETIME": "2026-03-19 16:00",
    "CUMULATION_PERIODS": "1hr,3hr,6hr,12hr,24hr",
    "HINDCASTING_MODE": true,
    "SEARCH_DATE": "2026-03-15",
    "RUN_TYPE": "06",
    "DELETE": true
}
